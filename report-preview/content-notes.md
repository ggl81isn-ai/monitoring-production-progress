# 週次報告（アプリ制作）— コンテンツ方針メモ

Notion連携前のモック用。実データに差し替える際は [index.html](index.html) の `id` / `data-field` を参照。

## Tailwind CSS

- スタイルは **Tailwind ユーティリティ**（HTMLの `class`）が主体。[src/input.css](src/input.css) に `@tailwind` と、**`data-*` 切替だけ** `@layer components` で記述。
- **ビルド**: `report-preview` で `npm install` のあと `npm run build:css` → [tailwind.css](tailwind.css) が生成される。HTMLは `tailwind.css` を読み込む。
- **監視**: `npm run watch:css` で保存のたびに再生成。
- 旧 `styles.css` は廃止（Tailwind移行済み）。

**レイアウト（v11・Tailwind）**: ルートは **`<main data-report-root>`**。ヘッダーの直後に **今週の進捗（信号）**、続けて **マイルストーン（横タイムライン4段）**。各 `section` は `aria-labelledby` と見出し `id` で対応。信号の3灯は **`aria-hidden`**、状態は **`#signal-status-label`** と **`#signal-caption`** で伝える。スクショでは「週次の結論 → 全体の現在地」の順で読める。

**サンプルプロジェクト**: モック本文は **自分用AIチャットボット** 制作を想定したダミー。別テーマにするときは `data-project` と各 `data-field` の文言を差し替える。

## LINEスクショ前提

- 受け取り側は画像のため、`<details>` などの開閉UIは使わない（現状のHTMLも開閉なし）。
- 信号の意味は **キャプション1行** に含めるか、別メッセージテキストで送る運用が確実。

## 報告に含める最小セット（案）

| ブロック | 目的 | 記載ルール（案） |
|----------|------|------------------|
| **今週の目標** | その週のコミットを固定する | 「今週の目標」見出し＋「・」始まりの箇条書き（本番HTMLはモック用の凡例行なし）。 |
| **マイルストーン** | プロジェクト全体の大区分 | **プロジェクトごとに4段の文言は固定**。詳細工程はNotion。いま何段目にいるかだけを `data-current-step`（1〜4）で示す（下記参照）。 |
| **今週のまとめ** | 事実の要約 | 2〜4文。完了タスク名・判断・ブロッカーがあれば1語。 |
| **次週アクション** | 次の一手のコミット | 箇条書き2〜4。 |
| **信号 + キャプション** | その週の進み具合 | `#signal-status-label` で見立て（色と一言）、`#signal-caption` は根拠のみ（見立て文と意味が重複しないようにする）。**マイルストーンとは独立**（下記）。 |

## 信号とマイルストーンの役割（独立）

- **信号（赤／黄／緑）**: **今週**の見立て。「今週立てた目標に対して、思った通り進んだか」。遅れ・未着手なら赤、など運用で決める。
- **マイルストーン4段**: **プロジェクト全体**のどの大きな段階にいるかの目印。**現在地を示すだけ**で、信号の色とは連動させない。

## `data-current-step` とは（平易な説明）

- HTMLの `#roadmap-visual` に付いている **`data-current-step="1"`〜`"4"`** は、「4つのマイルストーンのうち、**いま何番目の段階にいるか**」という**ただの番号**。
- **1**なら左端のラベルが「現在地」、**2**なら左から2番目、というように見た目が切り替わる。
- 自動化するときは、Notionなどの「いまのフェーズ」列の値を、毎週この数字（と `#milestone-caption` の一文）に写すイメージでよい。

## Notionに「タスクとして書いてあるだけ」で読み取り判断できるか

**いけそうです。** Notion APIでデータベースを取得し、「制作」タグ（または該当プロパティ）で絞り込み、**ステータス・期限・最終更新・チェックボックス**などを読めば、週次の信号や本文のたたき台を自動生成できます。

ただし運用で決めておくとブレません。

- **信号（赤黄緑）**: 「今週のゴール」がNotion上でどう表現されるか（専用プロパティ／今週期限のタスク集合／先週からの差分）を固定すると、機械判断の説明が一貫する。
- **マイルストーン（1〜4）**: タスク群から推論するより、DBに **フェーズ（select）** を1つ置き、週次ではその値を `data-current-step` に写すのがおすすめ。タスクだけだと粒度や抜けで誤判定しやすい。
- **週次のまとめ文**: 完全自動は難しくても、完了タスク名の列挙＋テンプレ1文は生成しやすい。人間が1文だけ直すハイブリッドも現実的。

## マイルストーン（横タイムライン）

- **山・イラスト**: 使わない。HTMLの **番号付き4ステップ＋接続レール** で表現。
- **コンテナ**: `#roadmap-visual`（`data-milestone-count="4"` はメモ用）
- **現在地**: `data-current-step="1"` … `"4"`。CSSで **左側＝完了（塗りつぶし）**、**該当＝現在地（リング）**、**右＝これから（線枠）** を切り替え。
- **ラベル**: `#milestone-label-1` … `4` のテキスト、または `.milestone-step__label` を生成差し替え。
- **キャプション**: `#milestone-caption`（例：`現在地：2. コア実装 — 進行中：…` のように簡潔に。FIX済みの列挙などは必須ではない）。
- モック開発時のみ、見出し直下に凡例を置くことはあったが **本番生成では出力しない**。

### マイルストーンが多いとき

- 週次スクショ用に **「意思決定・外部依存・リリース境界」** だけ残し、中間工程はNotionのタスク一覧に任せる。
- 4を超える場合は、HTMLを差し替えるか別テンプレート（5ノード用）を分けるより、**上位k件に集約**する運用がラク。

## Notion「制作」タグとの対応（次フェーズ）

- **対象**: タスクDBでプロパティ「タグ」等に `制作` が含まれる行。
- **信号の入力候補**（要決定）: 週内更新の有無 / 週内完了数 / 最終進捗からの経過日数 など。
- **マイルストーン**: Notionの「フェーズ」などを、週次HTMLでは **`data-current-step`（1〜4の数字）** と `#milestone-caption` の短文に写す想定。

## HTML 差し込みポイント一覧

| 要素 | id / 属性 | 内容例 |
|------|-----------|--------|
| ラベル | `#report-label` | 週次レポート・自分用 / AIチャットボット など |
| タイトル | `#report-title` | 例：自分用AIチャットボット — 進捗状況 |
| 対象週 | `#report-week-range` | 対象週 YYYY/MM/DD〜MM/DD |
| 今週の目標 | `#report-goal-summary` | 見出し「今週の目標」＋ `data-field="goal-item-*"` の行 |
| 今週の進捗見出し | `#signal-heading` | 今週の進捗 |
| 見立て1行 | `#signal-status-label` | 例：見立て：遅れ（赤） |
| マイルストーン見出し | `#milestone-heading` | マイルストーン |
| プロジェクト識別 | `[data-report-root]` の `data-project` | 任意（生成スクリプト用） |
| マイルストーン帯 | `#roadmap-visual` `data-current-step` | 1〜4 |
| マイル文言 | `#milestone-label-1` … `4` | 各ラベル |
| 現在地キャプション | `#milestone-caption` | 現在地の1文 |
| 信号 | `#traffic-state` `data-signal="red\|yellow\|green"` | 灯の強調はCSSが追従 |
| 信号説明 | `#signal-caption` | 判定根拠1行 |
| 今週の要約（先頭） | `#weekly-summary-lead` | スクショ向け短い結論（1〜2文） |
| 今週の詳細 | `#weekly-summary` | 補足段落 |
| 次週 | `#next-actions-list` 子 `data-action-id` | `ol` の `li` を生成で差し替え |

ルート: `<main data-report-root>` — スクショ範囲をこの要素に限定しやすい。

余白・最大幅は `tailwind.config.js` の `spacing.report*` / `maxWidth.report` を参照（`mb-report` など）。

## 信号の意味（運用メモ）

- **緑**: 今週の目標どおり（またはそれ以上）に進んだ
- **黄**: 軽い遅れ・一部未達だがリカバリ可能
- **赤**: 今週の見立てとして「思った通りに進んでいない」（遅れ・未着手・ブロッカーなど。定義は自分用で固定するとよい）

※根拠は `#signal-caption` に1行書くか、LINE本文で補足する。
