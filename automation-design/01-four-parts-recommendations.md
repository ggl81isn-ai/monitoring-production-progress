# 報告自動化の4パーツ設計（ベストプラクティス寄せ）

講義の枠組み（トリガー／ソース元／処理する場所／届ける先）に沿って、**週次の進捗報告**を想定したおすすめの組み方を整理する。前提として、ソースは **Notion（制作タスク等）**、成果物は **HTML レポート＋画像で LINE 通知** を想定する。

---

## タスクリスト（実装フェーズ用の確認項目）

- [ ] トリガー: 週次の時刻・タイムゾーン（JST か UTC か）を決める
- [ ] トリガー: 失敗時の再実行・通知方針（Actions の手動再実行、Slack 等）を決める
- [ ] ソース: Notion の対象 DB／データソース ID と、Bot に共有する手順を完了する
- [ ] ソース: 週の区切りに使うプロパティ（日付・ステータス・フェーズ）を固定する
- [ ] 処理: HTML テンプレートへの差し込み方式（ビルド時注入かランタイム置換か）を決める
- [ ] 処理: スクリーンショット対象（`data-report-root` 全体か要素単位か）を決める
- [ ] 届け先: LINE の Channel Access Token・ユーザー ID をシークレット管理する
- [ ] 届け先: 画像 URL 用の HTTPS 配信先（Artifacts・オブジェクトストレージ・一時 URL）を決める
- [ ] セキュリティ: API キー・トークンをリポジトリにコミットしない

---

## 1. トリガー（いつ動かすか）

### 推奨（このプロジェクト向け）

| 優先 | 手段 | 向いている理由 |
|------|------|----------------|
| 第一候補 | **GitHub Actions の `schedule`** | 週次 cron で再現性が高い。処理も同じリポジトリのワークフローにまとめやすい。 |
| 第二候補 | **Google Apps Script の時間主導トリガー** | Notion とスプレッドシートをまたぐ場合や、Google Workspace 内完結にしたい場合。実行時刻にジッタがある点に注意。 |
| 補助 | **Cursor Automations** | 「人がレビューしながらレポート文案を生成する」用途には有効。完全な定時バッチの代替としては料金・権限・再現性を要確認。 |

### ベストプラクティス（要点）

- **タイムゾーン**: GitHub Actions の `schedule` は **UTC** 基準。日本の「毎週月曜 9:00」などは cron を UTC に換算する（公式の `schedule` 説明を参照）。
- **GAS**: 時間主導トリガーは **作成者の権限** で動き、時刻に **ランダムなずれ** が入ることがある。厳密な定時が必要なら Actions 側を優先する。
- **Cursor**: スケジュール・GitHub・Slack・Webhook 等のトリガーが用意されている。エージェント向けの自動化であり、従来の「同じ入力なら同じバイト列の成果物」という意味のバッチとは設計思想が異なる。

---

## 2. ソース元（データの出どころ）

### 推奨

| ソース | 向いているケース |
|--------|------------------|
| **Notion API（データソース照会）** | 制作タスク・フェーズ・今週の目標などを DB 化している場合。 |
| **Git リポジトリ** | コミット数・差分・タグから「実装進捗」を補足する場合（任意）。 |

### ベストプラクティス（要点）

- **インテグレーション接続**: Notion の DB／ページは **インテグレーションに共有**しないと API で取得できない（404 になる）。
- **API バージョン**: Notion はデータベースとデータソースのモデルが更新されている。**「Query a data source」** 等、現行バージョンの Reference に合わせる。
- **フィルタ**: 週次なら「今週の日付範囲」「ステータスが進行中」など、**サーバー側フィルタ**で取得件数を抑える。

---

## 3. 処理する場所（加工・レポート化）

### 推奨

| 構成 | 内容 |
|------|------|
| **GitHub Actions 上のジョブ** | Notion から JSON 取得 → テンプレート HTML にマージ → **Playwright** で PNG 化 → 配信ステップ。 |

### ベストプラクティス（要点）

- **再現性**: 同じコミットのテンプレート＋同じ週の Notion スナップショットで、成果物が追跡できるようにする。
- **HTML→画像**: Playwright の **`page.screenshot`**（`fullPage` や要素指定）を使う。ヘッドレスブラウザは CI 用イメージと相性がよい。
- **GAS に集約する場合**: Notion は UrlFetch、スプレッドシートは SpreadsheetApp、メールは GmailApp など **Google 圏内**で完結しやすい。HTML レンダリングや高品質スクショは Actions の方が楽なことが多い。

---

## 4. 届け先（完成品の行き先）

### 推奨（ユーザー方針: LINE）

| 手段 | 内容 |
|------|------|
| **LINE Messaging API（Push）** | 週次で自分（または指定ユーザー）へプッシュ。テキスト要約＋画像メッセージの組み合わせが現実的。 |

### ベストプラクティス（要点）

- **Push**: `POST https://api.line.me/v2/bot/message/push`（Channel Access Token 必須）。
- **画像メッセージ**: **オリジナル用 URL とプレビュー用 URL の2つ** を指定する。どちらも **HTTPS（TLS 1.2 以上）**。LINE が取得できるよう、**認証なしで読める公開 URL** か、短命トークン付き URL などの設計が必要（社内限定 URL だけだと LINE サーバから取得不可）。
- **トークン**: Channel Access Token は **GitHub Encrypted secrets** 等で保持し、ログに出さない。

### 代替（講義の例）

- **Slack Incoming Webhook / chat.postMessage**: 社内が Slack なら運用しやすい。
- **メール（Resend 等）**: HTML メールまたは画像添付。配信ドメイン認証（SPF/DKIM）を公式に従う。
- **パブリック URL（Surge 等）**: 「リンクを共有」型の届け方。LINE に載せる画像 URL としても使えるが、アクセス制御は別設計。

---

## 推奨アーキテクチャ（1本の例）

1. **トリガー**: GitHub Actions `schedule`（週1、UTC cron）。
2. **ソース**: Notion API で対象データソースを Query。
3. **処理**: Node または Python ジョブで `report-preview/index.html` 相当を生成 → Playwright で PNG。
4. **届け先**: 画像を HTTPS でホスト（Actions Artifact だけでは LINE が取れない場合が多いので、S3・R2・GitHub Pages の一時デプロイ等を検討）→ LINE Push で image message を送信。

この分割により、講義の4パーツがそれぞれ **明確な責務** と **公式ドキュメントの置き場所** を持つ。
